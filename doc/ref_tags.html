Value Meta Labels

Variables and parameters have types or subtypes. The values that are stored in them can have optional data meta labels. The labels are names that are attached to a value and follow the values as they move through the script. Every value will have a set of tags, just like in the previous examples. Each variable, function parameters and other places that store values will also store the tags along with them.

There are different categories of value meta labels.

The Meta Statement

Unit-of-Measure Labels

Units of measure is a description of what a number represents. If a number is 500, what is it measuring? 500 of what? It could be temperature, distance, inventory stock, money or something else. The description names the things you are measuring. As I recall, there was a high-profile case where a NASA lander crashed into the surface of Mars because the North American team used American units while the European team used metric.

As an example, you should be able to declare a value in its currency:

   north_american_sales := 50_000 tagged us_dollars;
   european_sales       := 50_000 tagged euros;
   total_sales          := north_american_sales + european_sales; -- ERROR
                                                                ^ In the expression, the left value tag us_dollars is not compatible with
   the right value tag of euros

This is sometimes described as "comparing apples to oranges". You cannot add U.S. dollars to directly to Euros because they are different units.

Units of measure may be applicable to strings, even though strings do not measure anything per se. For example, I'm considering:

   company_name := "Sales Co" tagged english;

While you could create separate variables with strong typing for U.S. Dollars and Euros, it's not practical for a financial company working with 100 currencies, with values that are continually changed at run-time, or with projects that have memory constraints.

Policy Meta Labels

While units-of-measure is useful, my main objective is securing personal and financial data. I want to provide a safety net to stop sensitive data from being leaked.

Tags can track security policies. Each file is assigned policies that they support. Each value is assigned the data policy of the value. Values can only be written to files supporting the correct policy. For example, what if I want to label a credit card number with PCI, the security standard for the payment card industry?

   credit_card := "1234567890" tagged policy pci;
   put( "My credit card is " );
   put_line( credit_card );
                       ^ In put_line, the expression policy pci is not compatible with standard_output
(policy public)

In this case, a value under the PCI policy cannot be written to an unsecured public file like standard output. Even if a malicious threat actor manipulated a script to write such a value, the language will refuse to do so.

Unlike units-of-measure, policies will "bleed" or "taint" derived values.

   credit_card := "1234567890" tagged policy pci;
   checksum := strings.md5( credit_card );

The Authorize Statement

For policy meta labels, the label cannot be assigned unless the script is authorized to use the meta label.

  authorize policy pci begin
    credit_card := "1234567890" tagged policy pci;
 end;
