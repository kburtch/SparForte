Value Meta Tags

NOTE: This feature is under construction. The features are not complete and may change.

Variables and parameters have types or subtypes. The values that are stored in them can have optional data meta tags. The labels are names that are attached to a value and follow the values as they move through the script. Every value will have a set of tags, just like in the previous examples. Each variable, function parameters and other places that store values will also store the tags along with them.

There are different categories of value meta tags.

The Meta Statement

The meta statement defines new meta tags. A top-level meta tag is created with "is new meta".

   meta uom_name is new meta; -- a new unit-of-measure
   meta policy policy_name is new meta; -- a new security policy

Meta labels can also be abstract. These can be declared but not tagged to values.

   meta uom_name is new abstract meta;

New labels can be extended from existing labels.

   meta distance is new abstract meta;
   meta kilometers is new distance;
   meta miles is new distance;

Tagging Values

Meta tags are attached to a value with the tagged keyword. Tagged can be used on numeric literals, string literals, backquoted literals or enumerated type items.

  • 30 tagged kilometers
  • 50.0 tagged us_dollars
  • "company name" tagged utf8_english
  • `ls -l` tagged policy unchecked_data
  • apple tagged policy confidential -- where apple in an enumerated item

The value in variables cannot be tagged. You can only tag values at their point of origin. In x := y + 5, the "5" can be tagged but "y" cannot be tagged. The value of "y" must have originated somewhere earlier.

Enumerated items cannot be given a tag when they are defined in a type declaration.

Unit-of-Measure Labels

Units of measure is a description of what a number represents. If a number is 500, what is it measuring? 500 of what? It could be temperature, distance, inventory stock, money or something else. The description names the things you are measuring. As I recall, there was a high-profile case where a NASA lander crashed into the surface of Mars because the North American team used American units while the European team used metric.

As an example, you should be able to declare a value in its currency:

   north_american_sales := 50_000 tagged us_dollars;
   european_sales       := 50_000 tagged euros;
   total_sales          := north_american_sales + european_sales; -- ERROR
                                                                ^ In the expression, the left value tag us_dollars is not compatible with
   the right value tag of euros

This is sometimes described as "comparing apples to oranges". You cannot add U.S. dollars to directly to Euros because they are different units.

Units of measure may be applicable to strings, even though strings do not measure anything per se. For example, I'm considering:

   company_name := "Sales Co" tagged english;

While you could create separate variables with strong typing for U.S. Dollars and Euros, it's not practical for a financial company working with 100 currencies, with values that are continually changed at run-time, or with projects that have memory constraints.

Policy Meta Labels

While units-of-measure is useful, my main objective is securing personal and financial data. I want to provide a safety net to stop sensitive data from being leaked.

Tags can track security policies. Each file is assigned policies that they support. Each value is assigned the data policy of the value. Values can only be written to files supporting the correct policy. For example, what if I want to label a credit card number with PCI, the security standard for the payment card industry?

   credit_card := "1234567890" tagged policy pci;
   put( "My credit card is " );
   put_line( credit_card );
                       ^ In put_line, the expression policy pci is not compatible with standard_output
(policy public)

In this case, a value under the PCI policy cannot be written to an unsecured public file like standard output. Even if a malicious threat actor manipulated a script to write such a value, the language will refuse to do so.

Unlike units-of-measure, policies will "bleed" or "taint" derived values.

   credit_card := "1234567890" tagged policy pci;
   checksum := strings.md5( credit_card );

Meta Tags and Arrays

An array can contain values with any kind of meta tags. Each array item has its own meta tags. The array as a whole does not have meta tags because it does not have a value of its own.

This is allowed:

  some_array(1) := 15 tagged miles;
  some_array(2) := 25 tagged kilometers;

However, some subprograms using arrays may fail. For example, you won't be able to use stats.sum to add the values in an array unless the items all have the same unit-of-measure.

The Authorize Statement

SparForte is always running with a set of system security policies that it will enforce: anything policy that it not in SparForte's system policies will be rejected. These policies are managed at run-time, not compile-time, so that security policies are dynamic and a script can change policies when needed.

Security policies are a three-way relationship: the security policy of the source, the security policy of a target file and the current system security policies.

The default security policy set is empty, meaning security policies will not be enforced. This is for backward compatibility.

For policy meta tags, the label cannot be assigned unless the script is authorized to use the meta tag. The authorization expires at the end of the authorize block.

  authorize meta policy pci begin
    credit_card := "1234567890" tagged policy pci;
  end meta pci;

However, this will fail because policy gdpr is not the same as policy pci:

  authorize meta policy gdpr begin
    credit_card := "1234567890" tagged policy pci;
  end meta gdpr;

You cannot authorize unit-of-measure tags.

Assigning Meta Policies to Standard Files

Policies can be assigned to standard input, standard output or standard error using pragma meta_tag. Once a policy has been set, it cannot be changed.

  pragma meta_tag( standard_input, unchecked );
  pragma meta_tag( standard_output, pci );

In this example, every value read from standard input will have a policy of "unchecked". Every value written to standard output must have a policy of "pci".

  s : string;
  s := get_line;
  put_line( s ); -- error: policy unchecked is not policy pci

In this example, the value of variable s has the unchecked policy label. Since unchecked is not the expected policy, the put_line will fail.

User files inherit the policy of their pathnames.

   credit_card_file : file_type;
   credit_card_file_path : constant string := "cc.dat" tagged pci;
   open( credit_card_file, credit_card_file_path ); -- credit card file will have policy pci

In this example, any value written to credit_card_file must have a pci policy tag.
